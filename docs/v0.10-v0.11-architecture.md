## 1. Problem Recap

Three coupled assumptions that break for real-world users:

| Assumption | Reality |
|---|---|
| Config at workspace root | Multi-root workspaces, single-project folders, no-folder windows |
| `basename(cwd)` is unique | `packages/api` and `services/api` both resolve to `api` |
| Terminals auto-populate with usable names | Users with 15 projects across scattered directories need guided setup |

**Who hits this:** Windows users with `C:\Users\...\Documents\Claude Code\<project>` layouts, monorepo users, multi-root workspace users, anyone whose projects aren't siblings under one parent.

**Key invariant:** the config key (slug) = the terminal name = `CLAUDELIKE_BAR_NAME` env var = the status filename. The `path` field is the canonical identity; the slug is the human-friendly handle derived from it.

## 3. Config File Design

### 3.1 Location

**Single file:** `~/.claude/claudelike-bar.jsonc`
- Lives alongside `settings.json` and hooks — the user's Claude home directory.
- Single file across all workspaces. Opening any VS Code window loads the same config.
- The hook, statusline, and setup wizard already know `~/.claude/` via `claudePaths.ts`.
- No merge logic, no write-routing decisions, no "which file am I editing?" confusion.

**Per-workspace overrides** are deferred as a future feature ([#8](https://github.com/aes87/claudelike-bar/issues/8)). Ship simple, add the layer when someone asks for it.

### 3.2 Format

Same JSONC as today. New field `path` is the only structural addition to `TerminalConfig`:

```jsonc
{
  "mode": "passive-aggressive",
  "sortMode": "auto",
  "debug": false,
  "claudeCommand": null,

  "terminals": {
    "client-api": {
      "path": "/home/user/work/client-a/api",
      "cwd": "/home/user/work/client-a/api",
      "command": "claude",
      "color": "yellow",
      "icon": "server",
      "nickname": null,
      "autoStart": true
    },
    "personal-api": {
      "path": "/home/user/personal/api",
      "cwd": "/home/user/personal/api",
      "command": "claude",
      "color": "green",
      "icon": "server",
      "nickname": "my api",
      "autoStart": false
    }
  }
}
```

**`path` vs `cwd`:**
- `path` = canonical project identity. Used for matching hook status updates to tiles.
- `cwd` = working directory passed to `createTerminal({cwd})`. Defaults to `path` when unset.
- Separate because a user might want `cwd` to be a subdirectory of `path` (e.g., `path: /repo`, `cwd: /repo/packages/api`).
- In practice, most users set them to the same value or just set `path` and let `cwd` default.

**`path` when absent:**
- Backwards compat: if a terminal has no `path`, the extension falls back to name-based matching (3-tier exact/alias/normalized). This is the v0.9.x behavior.
- The setup wizard always sets `path`. Manual config entries can omit it.

### 3.3 Config read/write responsibility

All reads and writes target the single global file.

| Operation | Who |
|---|---|
| Read on activation | `ConfigManager` constructor |
| Write on terminal discovery | `ConfigManager.ensureEntry()` |
| Write on drag-reorder | `ConfigManager.applyDragOrder()` |
| Write on color/nickname change | `ConfigManager.setColor()` etc. |
| Write on setup wizard | Wizard command handler |
| Write on migration | Migration code (one-time, copies workspace → global) |

### 3.4 ConfigManager changes

Current `ConfigManager` reads from the workspace root. New version reads from `~/.claude/`:

```typescript
class ConfigManager {
  private configPath: string;  // ~/.claude/claudelike-bar.jsonc
  // Everything else stays the same — single file, single source of truth.
}
```

**Change:** the constructor switches from `workspaceFolders[0]` to `claudeDir()`. The watcher watches the new location. All external APIs (`getTerminal()`, `getAll()`, `getMode()`) are unchanged. Callers see no difference.

**Concurrent writes:** read-merge-write with retry. Before writing, re-read the file from disk, merge in-memory changes over the disk state, then atomic write. Handles two VS Code windows adding terminals simultaneously without losing one addition.

## 4. Project Identity Model

### 4.1 Slug

The config key is a **user-assigned slug**: a short, unique, human-readable name for a project. Examples: `client-api`, `personal-api`, `mushroom-tek`, `vscode-ext`.

**Slug rules:**
- Lowercase alphanumeric + hyphens. No spaces, no special chars.
- Must be a valid JSON key and a valid filename (status files are `{slug}.json`).
- Must be unique within the user's config.

**Slug derivation (in setup wizard):**
1. Start with `basename(path)` lowercased: `/home/user/work/client-a/api` → `api`
2. If collision detected (another terminal already has that slug), prepend the parent: `client-a-api`
3. If still colliding, prepend grandparent: `work-client-a-api`
4. User can always override with a custom name.

**Slug function (shared between extension and hook):**
```typescript
function deriveSlug(absolutePath: string, existingSlugs: Set<string>): string {
  const segments = absolutePath.split(/[/\\]/).filter(Boolean);
  for (let depth = 1; depth <= Math.min(segments.length, 4); depth++) {
    const slug = segments.slice(-depth).join('-').toLowerCase().replace(/[^a-z0-9-]/g, '-');
    if (!existingSlugs.has(slug)) return slug;
  }
  // All depths collide — append a numeric suffix.
  const base = segments.slice(-3).join('-').toLowerCase().replace(/[^a-z0-9-]/g, '-');
  for (let i = 2; i <= 99; i++) {
    const candidate = `${base}-${i}`;
    if (!existingSlugs.has(candidate)) return candidate;
  }
  throw new Error(`Cannot derive a unique slug for ${absolutePath}`);
}
```

### 4.2 Path matching

When a hook status file arrives with a project name, the extension resolves it:

1. **Exact slug match** — config key === incoming project name. Score 3.
2. **Path match** — incoming `cwd` (from status file or hook extra data) matches a terminal's `path`. Score 2.5.
3. **Existing 3-tier** (exact name / projectName alias / normalized). Scores 2/1.5/1. Backwards compat.
4. **No match** — logged, ignored.

The hook sets `CLAUDELIKE_BAR_NAME` to the slug for auto-started terminals, so path 1 is the hot path. Path 2 catches manually-opened terminals where the user navigated to a registered project directory.

### 4.3 Status file namespacing

**Current:** `{tmpdir}/claude-dashboard/{basename}.json` — collision-prone.
**New:** `{tmpdir}/claude-dashboard/{slug}.json` — collision-free for auto-started terminals (slug comes from CLAUDELIKE_BAR_NAME env var).

**Manual terminals (no env var):** hook falls back to `basename(cwd)`. If the basename collides with a slug, the file is shared — same behavior as today. The setup wizard mitigates this by encouraging auto-start for all projects.

## 5. Hook Changes

### 5.1 Minimal

The hook (`dashboard-status.js`) needs **zero changes** for the core identity fix:

- Auto-started terminals already get `CLAUDELIKE_BAR_NAME=slug` via `createTerminal({env})`.
- The hook's priority chain: `CLAUDELIKE_BAR_NAME` → `basename(cwd)` → `'unknown'`.
- Status file is written as `{slug}.json` when the env var is set.

### 5.2 Optional enhancement (v0.11)

For manually-opened terminals, we could have the hook attempt a path lookup:

1. Read `~/.claude/claudelike-bar.jsonc` (or a lightweight index file).
2. Check if `cwd` matches a registered `path` entry.
3. If yes, use that slug instead of `basename(cwd)`.

**Trade-off:** the hook fires 4+ times per Claude turn. Reading a config file on every fire adds ~1ms I/O. We could cache (read once, memoize by `cwd`), but the hook is a separate process on each fire (Claude forks it). Memoization doesn't persist.

**Alternative:** write a small index file `~/.claude/claudelike-bar-paths.json` that maps `path → slug`. The hook reads this (tiny, fast) instead of the full JSONC config. The extension regenerates the index whenever it writes the config.

**Recommendation:** defer to v0.11.x. For v0.10, the env-var path handles auto-started terminals (the primary use case), and manual terminals fall back to basename (acceptable).

## 6. Setup Wizard UX

### 6.1 Approach: Multi-step QuickPick

VS Code's `createQuickPick()` API supports multi-step flows natively. Microsoft's [quickinput-sample](https://github.com/Microsoft/vscode-extension-samples/blob/main/quickinput-sample/src/multiStepInput.ts) provides a reference `MultiStepInput` helper class.

**Why QuickPick over webview:**
- Native VS Code look and feel — no custom CSS, no webview security considerations.
- Each step is independently testable (mock `showQuickPick` return value, assert next step's inputs).
- Works identically on all platforms (webview rendering can vary).
- Keyboard-navigable by default.

**Why not walkthroughs:**
- Walkthroughs are declarative (Markdown + media) — they can't run multi-step logic.
- Good for documentation ("here's what Claudelike Bar does") but not for "pick your folders and configure them."
- Can coexist: a walkthrough page can link to the wizard command.

### 6.2 Wizard flow: "Claudelike Bar: Set Up Projects"

```
Step 1/5: Pick project folders
┌──────────────────────────────────────────┐
│ Select the folders where your Claude     │
│ projects live.                           │
│                                          │
│ ● Browse for folders...      (action)    │
│ ● Scan workspace folders     (action)    │
│ ● Import from existing config (action)   │
└──────────────────────────────────────────┘
```

**"Browse for folders"** → opens `showOpenDialog({ canSelectFolders: true, canSelectMany: true })`. User picks N folders. Each becomes a project entry.

**"Scan workspace folders"** → reads `vscode.workspace.workspaceFolders`, lists their immediate children as candidate projects. User checkmarks the ones they want. Works for the common `~/projects/<name>` layout.

**"Import from existing config"** → reads the workspace-local `.claudelike-bar.jsonc` (if it exists) and imports its terminal entries into the global config, deriving `path` from `cwd` where available.

```
Step 2/5: Name your projects
┌──────────────────────────────────────────┐
│ Confirm or edit the name for each        │
│ project. Names must be unique.           │
│                                          │
│ /home/user/work/client-a/api             │
│   → client-a-api                    [✏️]  │
│                                          │
│ /home/user/personal/api                  │
│   → personal-api                    [✏️]  │
│                                          │
│ /home/user/hobbies/martha-tek            │
│   → martha-tek                      [✏️]  │
└──────────────────────────────────────────┘
```

Slugs auto-derived via the `deriveSlug` function. User can click the edit icon (or select an item) to rename. Collision detection is live — renamed slugs are checked against the existing set.

```
Step 3/5: Assign colors
┌──────────────────────────────────────────┐
│ Pick a color theme for each project.     │
│ (You can change these later in the       │
│ config file or by telling Claude.)       │

│ personal-api:   ○ cyan  ● green  ...     │

└──────────────────────────────────────────┘
```

Simplified color picker — QuickPick items with color indicators. Or skip this step and auto-assign from the taxonomy (user can customize later).

```
Step 4/5: Choose startup command
┌──────────────────────────────────────────┐
│ What command should run when each         │

```

Single choice applied to ALL projects (can be overridden per-project later). Most users want the same command everywhere.

```
Step 5/5: Review and confirm
┌──────────────────────────────────────────┐
│ Ready to configure 3 projects:           │
│                                          │
│   client-a-api (cyan)                    │
│     /home/user/work/client-a/api         │
│     claude                               │
│                                          │
│   personal-api (green)                   │
│     /home/user/personal/api              │
│     claude                               │
│                                          │
│   martha-tek (magenta)                   │
│     /home/user/hobbies/martha-tek        │
│     claude                               │

│ ~/.claude/claudelike-bar.jsonc           │
│                                          │
│ [Confirm]  [Back]  [Cancel]              │
└──────────────────────────────────────────┘
```

On confirm: writes the global config, installs hooks (if not done), optionally installs statusline (if not configured). Shows a summary notification.

### 6.3 "Register Project" (single-add)

Palette command: "Claudelike Bar: Add Project"

1. `showOpenDialog({ canSelectFolders: true, canSelectMany: false, title: 'Select project folder' })`
2. Auto-derive slug, let user edit via `showInputBox`.
3. Auto-assign color (next unused from palette, or ask).
4. Write to global config.

### 6.4 Discoverability

- **First-run:** if no config exists anywhere (global or workspace), show a notification: "Claudelike Bar detected no project configuration. Set up your projects now?" → [Set Up Projects] / [Later]
- **Palette:** both "Set Up Projects" and "Add Project" commands registered.
- **Walkthrough (v0.11.x):** `contributes.walkthroughs` in package.json with steps explaining the bar, linking to the wizard command.

## 7. Migration Strategy

### 7.1 Workspace-local → global (one-time, v0.10.x)

On extension activation:
1. Check if `~/.claude/claudelike-bar.jsonc` exists.
2. If not, check if workspace-local `.claudelike-bar.jsonc` exists.
3. If workspace-local exists:
   a. Copy it to `~/.claude/claudelike-bar.jsonc`.
   b. For each terminal entry: if it has `cwd` but no `path`, copy `cwd` → `path`.
   c. Show notification: "Claudelike Bar: migrated your config to ~/.claude/claudelike-bar.jsonc. The workspace file is now an optional override."
   d. Do NOT delete the workspace file — user may want it as an override.
4. If neither exists: first-run path (trigger wizard or hooks-only install).

### 7.2 Slug derivation for migrated entries

Existing config keys are terminal display names (e.g., `"VS Code Enhancement"`, `"3d-printing"`). These may not be valid slugs (spaces, caps).

Migration normalizes keys:
- `"VS Code Enhancement"` → `"vscode-enhancement"` (lowercase, spaces → hyphens)
- `"3d-printing"` → `"3d-printing"` (already valid)
- `"Vault Direct"` → `"vault-direct"`

The original key is preserved as `projectName` alias for backwards-compat matching:
```jsonc
"vscode-enhancement": {
    "projectName": "VS Code Enhancement",
    "path": "/workspace/projects/vscode-enhancement",
    ...
}
```

### 7.3 No-config migration (truly fresh install)

New users who never had a workspace config:
1. Extension activates, finds no config.
2. Triggers first-run notification: install hooks + offer setup wizard.
3. If user picks "Set Up Projects", wizard runs.
4. If user picks "Later", hooks install, terminals auto-populate the global config as they open (same as today but writing to `~/.claude/`).

## 8. Onboarding Redesign

### 8.1 Current flow (v0.9.x)

```
Extension activates
  → is setup complete?
  → NO: show notification "Install hooks?"
    → "Install": run hooks + statusline
    → "Show me the hooks": open browser
    → "Later": do nothing
```

### 8.2 New flow (v0.10+)

```
Extension activates
  → does global config exist?
  │
  ├─ YES: load config, start normally.
  │   └─ does workspace override exist? merge it.
  │
  └─ NO: does workspace-local config exist?
      │
      ├─ YES: migrate to global (§7.1). Start normally.
      │
      └─ NO (truly fresh):
          → are hooks installed?
          │
          ├─ NO: show notification:
          │   "Claudelike Bar needs hooks to track terminal status.
          │    Set up your projects now?"
          │   → [Set Up Projects]: wizard (installs hooks as step 0)
          │   → [Install Hooks Only]: hooks + statusline, skip wizard
          │   → [Later]: do nothing
          │
          └─ YES (hooks but no config — unusual):
              → show notification:
              "You have hooks but no project configuration.
               Set up your projects?"
              → [Set Up Projects]: wizard
              → [Later]: terminals auto-populate as opened
```

### 8.3 Returning-user flow

User has a working config, reinstalls the extension (or upgrades):
- Hooks are re-checked and updated if events are missing.
- Config is already in `~/.claude/` — loaded immediately.
- No wizard, no notifications (the `SETUP_PROMPTED_KEY` flag is preserved in `globalState`).

### 9.3 Wizard testability

The wizard is decomposed into pure functions + thin VS Code API wrappers:

```typescript
// Pure function — takes user selections, produces config entries.

function buildProjectEntries(
  folders: string[],
  existingSlugs: Set<string>,
  colorAssignment: Record<string, string>,
```

### 9.4 Read-merge-write concurrency testability

The concurrent-write strategy is: re-read file → merge in-memory delta → atomic write. Testable via:
1. Write a file with known state.
2. Make an in-memory change (ensureEntry).
3. Before the debounced write fires, modify the file on disk (simulating another window).
4. Assert the final file contains BOTH changes.

### 9.5 Migration testability

Migration is a pure transform on a config object + filesystem:

```typescript
function migrateWorkspaceToGlobal(
  workspaceConfig: ConfigFile,
): { globalConfig: ConfigFile; slugRenames: Map<string, string> };
```

Tests verify: slug normalization, path derivation from cwd, projectName alias preservation, collision handling.

### 9.6 CI

Existing CI (3-OS vitest + setup.sh smoke) continues unchanged. New tests are vitest and run in the same matrix. @vscode/test-electron is deferred to v0.11.x and would need a separate CI job with `xvfb-run` on Linux.

## 10. Phased Implementation

### Phase 1: Path-based identity (v0.10.0)

**Goal:** fix the collision bug and lay the foundation for global config.

**Deliverables:**
1. Add `path` field to `TerminalConfig` interface.
2. `deriveSlug()` function in a new `src/slug.ts` module.
3. Status file naming: when `path` is set on a terminal, use slug-from-path for the status file instead of `basename(cwd)`. Hook already uses `CLAUDELIKE_BAR_NAME`, which the extension already sets from the config key — so this is mostly about the config key being a proper slug.
4. "Register Project" command: single folder picker → derive slug → add entry with `path` + `cwd` + default `command: "claude"` + auto-assigned color.
5. `cwd` defaults to `path` when unset in `getAutoStartTerminalOptions()`.
6. Path-based matching tier (2.5) added to `terminalTracker.matchScore()`.
7. Tests: slug derivation (collision handling, edge cases), path matching, register-project command, cwd-defaults-to-path.

**What ships:** collision fix for auto-start terminals, "Register Project" command, `path` as first-class identity. Config stays in workspace root for now.

**Test checkpoint:** 15+ new tests, all existing 196 pass, CI green.

### Phase 2: Global config + migration (v0.10.x)

**Goal:** move config to `~/.claude/`, auto-migrate from workspace-local.

**Deliverables:**
1. `ConfigManager` reads/writes `~/.claude/claudelike-bar.jsonc` (single file).
2. Read-merge-write concurrency: re-read disk before each write, merge in-memory changes over disk state, atomic write. Handles two VS Code windows.
3. Migration logic: workspace → global, slug normalization, `projectName` alias preservation.
4. Migration notification: "Config migrated to ~/.claude/claudelike-bar.jsonc."
5. `claudePaths.ts` gets a `globalConfigPath()` helper.
6. FileSystemWatcher watches `~/.claude/claudelike-bar.jsonc` instead of workspace root.
7. Tests: migration transform (slug normalization, collision, alias), read-merge-write concurrency, round-trip integration.

**What ships:** config works regardless of workspace layout. Existing users auto-migrate. Multi-root workspaces supported. No merge logic — one file.

**Test checkpoint:** 20+ new tests, all existing pass, CI green.

### Phase 3: Setup wizard (v0.11.0)

**Goal:** guided onboarding for new users and project reorganization.

**Deliverables:**
1. `src/wizard.ts` — multi-step QuickPick wizard.
   - Step 1: folder selection (browse / scan workspace / import existing).
   - Step 2: slug assignment (auto-derived, editable, collision-checked).
   - Step 3: color assignment (auto or manual).

## 11. Risk Analysis

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| Migration corrupts existing config | Low | High | Atomic write (tmp+rename); keep workspace file intact; log migration actions |
| Slug collisions in edge cases | Low | Medium | 4-depth derivation + numeric suffix; user override in wizard |
| Global config breaks multi-user systems (shared `~/.claude/`) | Low | Medium | Config is per-user by definition (`~/.claude/` is `$HOME`-scoped); document for shared-account setups |
| QuickPick UX is too constrained for 15+ projects | Medium | Medium | Step 2 (naming) could be overwhelming with many items; consider a "confirm all defaults" shortcut |
| Hook reads config on every fire (v0.11 path-lookup) | Medium | Low | Index file is <1KB; or defer and keep basename fallback |
| Breaking change for users who scripted `.claudelike-bar.jsonc` path | Low | Low | Workspace file is left in place after migration; scripts targeting it still read valid JSONC (just stale). Migration notification tells user the new location. |
