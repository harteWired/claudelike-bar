# Claudelike Bar v0.9 — Richer State Awareness

Proposal. Goal: fix the "shows done when waiting for subagent" bug and opportunistically add the high-value signals from [the field report](https://github.com/harteWired/claudelike-bar/issues) that cost us nothing extra.

## The bug, in detail

Current state machine:
```
working → ready (Stop/Notification)
ready → waiting (60s timeout)
waiting/ready → done (user looked away, chill mode)
waiting/ready → ignored (user looked away, passive-aggressive mode)
```

We register 4 hook events: `PreToolUse`, `UserPromptSubmit`, `Stop`, `Notification`. Claude Code supports [26 events](https://code.claude.com/docs/en/hooks). The ones we're missing matter.

### Failure mode 1: Task tool subagents (transient)

User has a main agent that invokes the `Task` tool to spawn a short-lived subagent. Claude Code fires:

1. `PreToolUse` with `tool_name: "Task"` — main agent is about to delegate
2. `SubagentStart` — subagent begins
3. `SubagentStop` — subagent finishes
4. Main agent receives results, either continues or finishes

Today we only see (1) as a generic `PreToolUse`. The tile stays "working" during the subagent run because the main agent's `Stop` doesn't fire until everything completes. So this case isn't actually broken — the real issue is the next one.

### Failure mode 2: Agent Teams (persistent) — **the actual bug**

[Agent Teams](https://code.claude.com/docs/en/agent-teams) run multiple independent Claude processes. Each has its own turn lifecycle. When Teammate A finishes its turn but is conceptually *waiting* for Teammate B to produce input:

1. Teammate A's turn ends → `Stop` fires → Claudelike Bar tile → `ready`
2. 60s passes → tile → `waiting`
3. User glances at tile, sees amber, switches away → tile → `done` (chill mode) or `ignored` (passive-aggressive)
4. Teammate B eventually finishes, signals Teammate A, which resumes
5. But Claudelike Bar has "parked" Teammate A as `done` — the tile is wrong

Claude Code fires a [`TeammateIdle`](https://github.com/anthropics/claude-code/issues/23545) event specifically for this: *"runs when a teammate is about to go idle."* That's the exact signal we need but aren't listening for.

### Failure mode 3: API errors (silent)

When Claude's turn ends with a rate limit, billing error, or auth failure, we see a `Stop` (maybe) or nothing. Tile looks the same as a successful finish. User doesn't know anything broke. `StopFailure` fires in this case with an `error_type` matcher (`rate_limit`, `authentication_failed`, `billing_error`, etc.).

## What v0.9 ships

Three fixes, scoped tight so we can land them together without scope creep.

### 1. Multi-agent state tracking (the core fix)

**Register these events:**
- `SubagentStart` (any matcher) — subagent spawned
- `SubagentStop` (any matcher) — subagent finished
- `TeammateIdle` — Agent Teams teammate going idle

**Hook script changes (`dashboard-status.js`):**
- Extract `tool_name` from `PreToolUse` payloads (for the Task case, supplementary to Subagent events)
- Extract `agent_id` and `agent_type` from Subagent payloads
- Add new fields to status file: `subagent_count`, `teammate_idle`

**State machine changes (`terminalTracker.ts`):**
- Per-tile `pendingSubagents: number` counter
- Per-tile `teammateIdle: boolean` flag
- `SubagentStart` → increment counter, status stays `working`
- `SubagentStop` → decrement counter (floor at 0)
- `TeammateIdle` → set flag true, status stays `working` with modified label
- `Stop` transitions to `ready` **only** when `pendingSubagents === 0` **and** `!teammateIdle`
- `UserPromptSubmit` resets both (counter = 0, flag = false) — universal reset
- Next `PreToolUse` or `UserPromptSubmit` clears `teammateIdle` (teammate is back to work)

**UI (`webview.js`):**
- Subagent count badge when `subagent_count > 0`: *"Working (2 agents)"*
- Teammate-idle label: *"Waiting for teammate"*
- Both use the existing green working dot so status coloring stays consistent

### 2. Error state for StopFailure

**Register:** `StopFailure` with matchers `rate_limit`, `authentication_failed`, `billing_error`, `server_error`, `max_output_tokens`

**State machine:**
- New state: `error`
- Transitions: any → `error` on `StopFailure` (stickier than `ready` — an error isn't resolved by a 60s timer)
- `error` → `working` only on `UserPromptSubmit` (user's explicit retry)
- `error` survives user focus changes (doesn't transition to `done`/`ignored`)

**UI:**
- Red static dot (distinct from `ignored` which is red dashed pulse)
- Label carries the error reason: *"Error: rate limit"*, *"Error: auth failed"*, etc.
- Sorted above `waiting` in auto-sort — errors need attention first

### 3. Notification matcher awareness

Current Notification handling is a single undifferentiated bucket. Claude Code matches it by `notification_type`: `permission_prompt`, `idle_prompt`, `auth_success`, `elicitation_dialog`.

**State machine:**
- `Notification` with `permission_prompt` → `ready` with label *"Needs permission"*
- `Notification` with `idle_prompt` → `ready` with label *"Awaiting input"* (current default)
- `Notification` with `auth_success` → no state change, flash the tile briefly
- `Notification` with `elicitation_dialog` → `ready` with label *"MCP needs input"*

No new state needed — just richer labels on `ready`. Near-zero implementation cost.

### Not in scope for v0.9 (deferred)

- `SessionStart`/`SessionEnd` — adds "active"/"offline" states. Useful but our tile lifecycle is already tied to VS Code terminals, and `SessionEnd` doesn't mean the terminal closed. Could cause tile/terminal mismatch. Revisit after we see how the above lands.
- `PreCompact`/`PostCompact` — brief "compacting" flash. Nice but cosmetic.
- `TaskCreated`/`TaskCompleted` — task count badges for the `TaskCreate` tool. Separate concern from the agent-state bug. Users who care can add later.
- `PermissionRequest`/`PermissionDenied` — more granular than `Notification` matchers. If `permission_prompt` matcher isn't enough, upgrade later.
- `PostToolUseFailure` — "tool errored" state. Probably noise — users see the error in Claude's output anyway.
- `ConfigChange`/`CwdChanged`/`FileChanged` — unrelated to sidebar status.

## Implementation plan

Mostly additive — no state machine semantics change for existing users.

| File | Change | Scope |
|------|--------|-------|
| `hooks/dashboard-status.js` | Extract `tool_name`, `agent_id`, `agent_type`, `error_type`, `notification_type` | ~20 lines |
| `src/types.ts` | `StatusFileData` gets `subagent_count`, `teammate_idle`, `tool_name`, `error_type`, `notification_type` | ~10 lines |
| `src/setup.ts` | `HOOK_EVENTS` extended with `SubagentStart`, `SubagentStop`, `TeammateIdle`, `StopFailure` (still a single shared hook script — one merger, all events) | ~5 lines |
| `src/terminalTracker.ts` | New `error` status, `pendingSubagents` + `teammateIdle` per tile, state transitions | ~60 lines |
| `src/statusWatcher.ts` | Pass new fields through to tracker | ~10 lines |
| `media/webview.js` + `webview.css` | Red static dot for `error`, subagent badge, teammate-idle label | ~30 lines |
| `test/hook.test.ts` | Subagent events, StopFailure, Notification matchers | ~10 tests |
| `test/terminalTracker.test.ts` | Multi-agent state transitions, error stickiness | ~12 tests |
| `docs/HOOKS.md` | Document the 8 registered events (was 4), new status file fields | docs |

**Migration:** Existing v0.8 users get the new events added automatically when the extension next runs `runSetup()` (idempotent — adds missing, leaves present). The first-activation onboarding covers new users.

**Breaking changes:** None. Status file schema is additive. Existing state machine behavior preserved for users who don't use agents.

**Estimated size:** ~200 lines of production code + ~22 new tests. Single release, no phases.

## Open questions

1. **`tool_name` on every `PreToolUse`?** The hook script already runs for every PreToolUse. Extracting `tool_name` is cheap. Worth doing? Useful for future "currently running tool" labels. Probably yes — doesn't hurt.

2. **`teammateIdle` flag decay?** If `TeammateIdle` fires, tile stays in "working (waiting for teammate)" indefinitely until next PreToolUse/UserPromptSubmit. Is there a reasonable timeout? The docs don't say Claude Code fires anything else when the teammate resumes. Probably fine to wait forever — `UserPromptSubmit` always resets.

3. **Passive-aggressive mode behavior during `TeammateIdle`?** Currently a tile in `waiting` with the user's attention then-lost gets guilt-tripped ("Patiently judging you"). Should a `teammate_idle` tile still trigger that? No — the user didn't cause the wait. Honor the flag and skip the guilt trip.

4. **`error` state sorting priority?** Auto-sort order today: `waiting → ignored → ready → working → done → idle`. Insert `error` at the top? Between waiting and ignored? I'd put it at the top — errors need attention more than waiting.
