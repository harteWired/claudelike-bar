# Claudelike Bar v0.9.1 — The Rest of the Hook Events

Follow-up to [v0.9](./v0.9-proposal.md). v0.9 shipped the high-priority signals (multi-agent, error state, notification matchers). v0.9.1 picks up the deferred events from the field report where they earn their keep.

## Prioritization principle

Each new event must clear a user-visible bar: either prevent a wrong tile state, surface a currently-invisible state, or enable a workflow users would notice. Eventing for its own sake is noise.

## In scope for v0.9.1

### 1. `SessionStart` / `SessionEnd` — lifecycle awareness

**Problem:** Today, a tile exists because a VS Code terminal exists. If a user closes Claude (Ctrl+D) but leaves the terminal open, the tile keeps showing the last known status — which may be "Working (2 agents)" from a dead session. Conversely, if Claude starts, the first visible transition isn't until UserPromptSubmit — a brief wrong state.

**Fix:** Register both events. `SessionStart` transitions `idle → working` (or stays `working` — just signals life). `SessionEnd` transitions tile to a new `offline` state (grey dot, dimmed) until a new `SessionStart` or `UserPromptSubmit`.

**Complexity:** ~30 lines across tracker + webview + config.

**Risk:** `SessionEnd` fires on `clear`, `resume`, `logout`, `prompt_input_exit`, `bypass_permissions_disabled`, `other`. Most of these shouldn't gray the tile (`clear` / `resume` are mid-session). Matcher `logout` + `prompt_input_exit` are the true-end signals. Filter accordingly.

**New state:** `offline` — sorted below `idle` (not urgent), grey dot, no animation. Auto-cleared by any subsequent real event.

### 2. `PostToolUseFailure` — tool errors

**Problem:** Right now, `error` (StopFailure) catches API errors. But if Claude tries to run a bash command that returns non-zero, or an edit fails, the hook just sees `PreToolUse → Stop` with no error signal. The tile shows "Ready" despite the conversation being in a broken state.

**Fix:** Register `PostToolUseFailure`. Don't transition to the full `error` state (a tool failure isn't the same as a session-ending API error) — instead, add a transient warning flag on the tile ("Working (tool error)") that clears on the next successful `PreToolUse` or `Stop`.

**Complexity:** ~20 lines. New `toolError?: boolean` flag on tile.

**Risk:** Noisy. Most tool failures are recovered by Claude within the same turn (e.g. a typo in a bash command, Claude sees stderr, tries again). A flag that appears and disappears rapidly is worse than no flag. **Needs observation in practice before committing to this.** Consider a debounce: only show the indicator if the tool error persists past the next `PreToolUse`.

### 3. `PreCompact` / `PostCompact` — compaction flash

**Problem:** When Claude auto-compacts the conversation, there's a ~5-10s window where the terminal is unresponsive but the tile shows "Working". Users who know what compaction is don't need the indicator; users who don't need to know that Claude is busy with maintenance, not their task.

**Fix:** Register both. `PreCompact` sets a transient `compacting: true` flag with label "Compacting context...". `PostCompact` clears it. No state change — the underlying status stays `working`, just the label changes.

**Complexity:** ~15 lines. Pure label override.

**Risk:** Low. Compaction is fast; flag is short-lived. Worst case: a PostCompact never fires (crash mid-compaction) and the label stays stuck. Fallback: clear the flag on any other subsequent event.

## Deferred again (out of scope for v0.9.1)

### `TaskCreated` / `TaskCompleted`

The `TaskCreate` tool is about task-list management (like the one we use in this session). It's not a hook into subagent coordination. Users who use it heavily might want a task-count badge, but it's a niche use case and would bloat the tile. Revisit if requested.

### `PermissionRequest` / `PermissionDenied`

v0.9 already uses Notification matchers (`permission_prompt`) to label `ready` tiles as "Needs permission." `PermissionRequest` gives finer-grained payload (includes `tool_name`, `tool_input`) which would let us label more precisely: "Needs permission: Bash". That's nice but not critical.

### `ConfigChange` / `CwdChanged` / `FileChanged`

Not state-relevant for a terminal dashboard. `ConfigChange` might be interesting if we wanted tiles to visually react to `.claudelike-bar.jsonc` edits, but we already watch that file directly in the extension — no benefit.

### `InstructionsLoaded`

Fires whenever a `CLAUDE.md` loads. Could show "loading context" briefly but almost nobody would benefit from seeing this.

### `WorktreeCreate` / `WorktreeRemove`

Already handled by VS Code's own terminal lifecycle events (open/close). Registering these would duplicate signals.

### `Elicitation` / `ElicitationResult`

MCP-specific. Already covered by `Notification` with `elicitation_dialog` matcher (added in v0.9) which labels the tile "MCP needs input".

### `TeammateIdle` (already in v0.9)

Already handled in v0.9 for the Agent Teams case.

## Implementation plan

Same batch pattern as v0.9. Three batches with `/review` between each.

### Batch 1: Events + hook script + types
- Register `SessionStart`, `SessionEnd`, `PostToolUseFailure`, `PreCompact`, `PostCompact` (5 new events; HOOK_EVENTS goes from 8 to 13)
- Extract `source` (SessionStart matcher), `compaction_trigger` (PreCompact matcher)
- New signals in hook script: `session_start`, `session_end`, `tool_failure`, `compact_start`, `compact_end`
- Add `offline` to `SessionStatus`, add `toolError`, `compacting` fields to `TileData`

### Batch 2: State machine + webview
- `session_start` → clear `offline` if set, no other change
- `session_end` (filter matchers: only `logout`/`prompt_input_exit`) → tile → `offline`
- `tool_failure` → flag the tile (flag cleared on next successful PreToolUse or Stop)
- `compact_start` → set `compacting`, label override
- `compact_end` → clear `compacting`, revert label
- Webview: grey static dot for offline; amber border glow for tool error; label override for compacting

### Batch 3: Tests + docs + release
- ~12 new tests across offline, tool error, compact transitions
- HOOKS.md: 13 registered events, 8+ optional status-file fields
- Package v0.9.1, publish, tag, push

## Size estimate

Similar to v0.9 — ~150 lines of code, ~12 new tests. Smaller than v0.9 because the state machine scaffolding is already in place; we're adding variants to existing patterns.

## Open questions

1. **`offline` state display**: grey dot is uninteresting. Should it be a dashed border or a faded tile? Probably match "done" styling since the distinction is "user parked" vs "Claude left" — both are dim.

2. **Tool-error debounce length**: how long to hold the flag before clearing? Options: (a) clear on next PreToolUse (could be ms), (b) clear on next Stop (lasts the rest of the turn), (c) explicit timer (5s?). Probably (b) — the flag marks "a tool error happened this turn" which is the useful signal.

3. **Compacting flash duration**: should it persist for a minimum time? If PostCompact fires within 200ms of PreCompact, users barely see the label change. Maybe min-display 500ms.

4. **Should `session_end` respect `done` stickiness?** If user marked a tile done, and then the session ends — should we still show `offline`? Probably yes — `offline` is more informative than `done` and the user can re-park if they want.

## Timing

Depends on real-world feedback on v0.9:
- If v0.9 lands clean and users want more signal → ship v0.9.1 shortly after
- If v0.9 exposes any state-machine edge cases → fix those first as v0.9.x patches, defer v0.9.1
