Right-click any tile NEW

Mark as done parks a session you're not working with right now. The tile sinks to the bottom of auto-sort, its dot goes quiet, and — critically — background Stop / Notification / PreToolUse events from that Claude won't re-surface it or trigger passive-aggressive judgement. It stays parked until you submit a new prompt in that terminal; the next UserPromptSubmit is the only thing that un-parks it.

Clone terminal spawns a fresh copy of the same profile (cwd, shell, args). Set color picks a per-terminal theme override — click a swatch, no file editing needed. Kill terminal disposes it entirely.

WSL2 / SSH / Dev Containers / Codespaces v0.13.3

The extension now declares extensionKind: ["workspace", "ui"] — VS Code runs the Node.js side of the extension on the same host as Claude Code, so hooks register to the right ~/.claude/settings.json, status files land in the same os.tmpdir() the extension watches, and the config is one file (not two ships passing in the night). Local-only setups fall through to the "ui" kind and behave exactly as before.

Audio alerts v0.12

Optional chime when Claude is waiting on you. Two slots — one for end-of-turn (Stop event), one optional for mid-job permission prompts (Notification event) — so you can tell "done" apart from "blocked on approval" by ear. Off by default; bring your own MP3/WAV/OGG, drop into ~/.claude/sounds/, point the config at it. Right-click any tile to mute/unmute.

Launch any registered project v0.13

Set up a big config of registered projects with autoStart: false on the ones you don't want auto-launching. Then open them on demand: rocket icon in the sidebar header (or the Launch Registered Project palette command, or Launch another project… in any tile's right-click menu). QuickPick of every config entry not currently open. Same launch path as auto-start, no drift.

Register without opening v0.13

Register Project flow ends with a "Open terminal now / Register only" choice. Pick "Register only" to add the entry with autoStart: false and launch it whenever via the rocket icon. Build out a 15-project registry without spawning 15 terminals.

Custom color picker v0.13.2

Right-click a tile → swatch row gains a rainbow <input type="color"> at the end. Click → OS color picker → any hex value flows through the same setColor handler. Pairs with the existing CSS-color support: #hex, rgb(), hsl(), var(--anything).

"Switch to auto sort" v0.13.1

Once you've dragged tiles around, sortMode flips to "manual" and stays there. Right-click any tile when in manual mode for a one-click Switch to auto sort entry — flips it back without opening the config. The entry hides itself when you're already in auto.

Auto-start cwd pre-flight v0.13.1

If you've moved/renamed/deleted a project dir, the bar used to surface VS Code's modal "Starting directory does not exist" error per stale entry. Now: fs.existsSync check before createTerminal, missing entries skipped, one warning toast at the end with an Open Config action. No modal pile-up.

Drag & drop to reorder NEW

Grab any tile, drop it where you want. A blue rail shows the insertion point. The order is saved to .claudelike-bar.jsonc and survives window reloads.

One-click project add v0.11.5

Click "+" in the sidebar → pick a folder → name it → terminal opens with Claude running and a tile in the bar. No config editing, no reload. Sets autoStart: true so it reopens on next VS Code launch.

Debug log

Flip "debug": true in the config. The extension streams hook events, state transitions, and auto-start decisions to a Claudelike Bar output channel. The hook and statusline scripts also write {os.tmpdir()}/claude-dashboard/debug.log. Useful when a tile seems stuck.

Windows / PowerShell support v0.9.2

Auto-start no longer assumes bash. CLAUDELIKE_BAR_NAME flows through VS Code's createTerminal({env}) API — cross-platform, no export noise. Per-terminal shellPath lets you pin git-bash (or pwsh, or cmd) when your command needs a specific shell.

Standalone statusline with backup + restore v0.9.2

Context % comes from a completely-independent statusline script. Installing it is opt-in — if you already have one, a modal shows both the command being overwritten and the one being installed before you confirm. Your prior statusline backs up to ~/.claude/.claudelike-bar-statusline-backup.json; a Restore Previous Statusline palette command (or Claude, via chat) puts it back.

Path-based project identity v0.10

Projects are keyed by absolute path, not terminal name. Collision-resistant slug derivation means two projects both named api in different trees get distinct identities (client-a-api vs personal-api). No more basename conflicts in monorepos or multi-root workspaces.

Global config at ~/.claude/ v0.10

Config moved from the workspace root to ~/.claude/claudelike-bar.jsonc — one file across all workspaces, next to your hooks and settings. Workspace-local configs auto-migrate on first load. Two VS Code windows adding terminals simultaneously? Read-merge-write handles it.

Setup wizard v0.11

5-step QuickPick flow: pick your project folders (browse or scan workspace), confirm names with collision-resistant slugs, assign colors (auto round-robin or manual), choose a startup command, review and confirm. Run from the palette: Claudelike Bar: Set Up Projects.

First-run walkthrough + sidebar button v0.11

VS Code's native walkthrough API guides new users through setup. The sidebar "+" button registers a project and opens the terminal immediately — folder picker, name it, Claude starts. No reload needed.

Hook path-lookup index v0.11

Manual terminals (opened without auto-start) now resolve correctly. The extension writes a lightweight ~/.claude/claudelike-bar-paths.json index — the hook reads it to map cwd to the right project slug. Atomic writes, trailing-slash normalized.

Configure it by talking to Claude

The config file is .claudelike-bar.jsonc at ~/.claude/ — JSON with comments, auto-created when you first open a terminal. You can edit it by hand, but the easier path is to just tell Claude Code what you want. After installing, open a Claude terminal and say: "Walk me through configuring the Claudelike Bar."

You
Switch to passive-aggressive mode and rename yeet-to-prod to "deploy".
Claude
Done. Two edits to .claudelike-bar.jsonc:
"mode": "chill", "mode": "passive-aggressive", "terminals": { "yeet-to-prod": { "color": "yellow", "nickname": null, "nickname": "deploy", "autoStart": true } }
The extension picked them up live — no reload needed.
You
Auto-start world-domination, print-go-brrr, brain-dump, and spore-drive. Each should open in its project directory and launch Claude.
Claude
Four terminals configured with path and autoStart:
"world-domination": { "path": "/home/you/projects/world-domination", "autoStart": true, "command": "claude" }, "print-go-brrr": { "path": "/home/you/projects/print-go-brrr", "autoStart": true, "command": "claude" }, ...
The path sets the working directory — no cd needed, works on every platform. Next VS Code launch, all four open automatically.
You
sourdough-ops needs different flags — it should run Claude with the MCP plugin.
Claude
Added a per-terminal command override:
"sourdough-ops": { "path": "/home/you/projects/sourdough-ops", "color": "green", "autoStart": true, "command": "claude --mcp-config mcp.json" }
You
Something's weird — print-go-brrr is stuck on "Working" forever. Can you figure out what's wrong?
Claude
Turning on debug logging:
"debug": false, "debug": true,
Open the Claudelike Bar output channel (Ctrl+Shift+U) and run a prompt in print-go-brrr — I'll watch for hook events and state transitions — if the hook never fires or the project name doesn't match the tile, we'll see it in the trace.

Everything you see in these conversations is plain JSONC in ~/.claude/claudelike-bar.jsonc. Comments, section headers, and defaults are preserved by the extension's round-trip save — feel free to hand-edit. Or run the setup wizard for a guided start. But most days it's easier to just ask.
